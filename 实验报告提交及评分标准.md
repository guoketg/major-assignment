## 五、硬性扣分项

出现以下情况之一，相关评分项不得满分；严重时可判定综合实验不通过：

1. 没有接入真实大模型。
2. 没有智能体流程，只是普通问答、静态页面或规则脚本。
3. 没有 MCP 风格工具契约，或工具契约与实际调用不一致。
4. 提交真实密钥、令牌、账号、内部地址或未脱敏隐私数据。
5. 无法提供本地运行命令。
6. 没有自动化测试。
7. 没有失败路径、安全拒绝或错误修正记录。
8. 代码与报告明显不一致，或无法解释关键实现。
9. 报告未提供项目仓库地址，也未说明替代提交方式。

## 六、加分项

- Docker 路径完整跑通，并能说明容器与宿主机模型服务的关系：+4 分。
- 实际实现完整 MCP 服务或客户端调用，并能用日志证明智能体通过该协议调用工具：+5 分。
- 有较完整的评估样本集和结果统计，例如通过率、失败类型、改进前后对比：+4 分。
- 智能体日志可视化清楚，能展示目标、计划、工具调用、结果和失败出口：+4 分。
- 安全护栏设计深入，例如工具白名单、目录沙箱、敏感信息扫描、人工确认和审计日志组合使用：+4 分。
- 项目创意完整且有真实个人使用价值，功能虽小但闭环扎实：+3 分。

## 七、注意事项

1. 本综合实验默认个人完成。
2. 不要求公网部署，不要求课堂现场演示。
3. conda 是必做环境路径；Docker 是优先推荐路径，受设备条件限制时可用说明替代。
4. 真实大模型和智能体接入是硬性验收项。
5. 至少 1 个 MCP 风格工具契约是硬性验收项；完整 MCP 服务或客户端调用是加分项。
6. 使用 AI 编程智能体辅助开发是允许的，但同学本人必须负责验证、测试和解释。
7. 除综合实验报告外，项目材料推荐统一放在自己的 Git 仓库中管理；报告中写明仓库地址。
8. 报告和仓库中不要展示真实密钥、令牌、账号、内部地址、隐私文本或未脱敏日志。
9. 如果项目无法在授课老师机器上直接运行，应在报告中说明环境差异和替代验证证据。

## 八、常见问题

### 可以提交团队项目吗？

默认个人完成。如果授课老师允许多人合作，需要在报告中明确每个人的工作范围、提交记录和可验证贡献。

### 容器环境没跑通会不会直接不通过？

不会直接不通过。Docker 是优先推荐路径，conda 路径是必做路径。Docker 不可用时，需要说明 daemon、权限、网络、系统或设备限制，并提供 conda 跑通证据。

### 本地小模型回答质量一般怎么办？

本综合实验重点是智能体工程闭环，不要求模型回答达到商业系统水平。可以通过更清晰的提示词、检索增强、输出约束、人工确认和失败处理提升可用性。

### 没有在线接口账号怎么办？

可以使用本地 Ollama 小模型。只要能完成真实模型连通性测试，并在报告中说明模型名称和限制即可。

### 可以把 AI 编程智能体生成的代码直接提交吗？

不建议直接提交未经理解和验证的代码。可以使用 AI 编程智能体辅助开发，但必须通过测试、运行和安全检查，并在报告中说明自己做了哪些判断和修正。

### 必须做完整协议服务吗？

不强制。必须提交并实际使用至少 1 个 MCP 风格工具契约。能进一步实现完整 MCP 服务或客户端调用的项目，可以获得加分。
