## 摘要

本文设计并实现了一个基于多 Agent 协作的科研分析助手系统。该系统以 LangGraph 为工作流引擎，编排七个 AI Agent 节点，通过 DeepSeek 大模型 API 驱动，自动化完成文献调研、创新构思、实验设计等科研流程。系统采用 FastAPI 构建后端服务，React 构建前端界面，Docker 容器化部署，实现了流式输出、三层记忆管理、MCP 风格工具契约、会话持久化等核心功能。测试结果表明，系统各模块运行稳定，Agent 流水线可正确完成意图识别、任务分配、工具调用和结果综合的全流程。本文从系统架构、关键技术、测试验证等方面详细阐述了设计与实现过程，并讨论了词元成本、安全风险和未来改进方向。

**关键词**：AI Agent；LangGraph；大语言模型；科研辅助；多 Agent 协作

## 1 引言

### 1.1 研究背景与问题

在深度学习研究领域，科研人员面临文献调研耗时、创新构思困难、实验设计重复以及信息碎片化等多重挑战。传统的工作方式中，论文检索需要手动在学术搜索引擎上逐一搜索筛选，创新想法缺乏系统性的评估和整理，实验设计方案凭经验设计而没有规范化的模板，论文笔记、实验结果和创新想法则分散在不同的文件和工具中。

针对上述问题，本文提出并实现了一个多 Agent 协作的科研分析助手，通过 LangGraph 编排多个具有不同专业能力的 AI Agent，构建自动化的科研工作流。系统能够根据用户输入自动识别意图、分解任务、调用工具完成信息检索，并生成结构化的分析报告。

### 1.2 项目目标

[图片：项目整体流程示意图]

本项目的核心目标包括以下四个方面：第一，能够调用真实大模型进行智能对话和任务处理；第二，智能体能够自动搜索学术论文并分析整理；第三，支持多轮对话和上下文记忆；第四，实现 Docker 容器化一键部署。

## 2 系统架构设计

### 2.1 整体架构

系统采用前后端分离架构。前端为 React 单页应用，提供聊天界面、Agent 可视化面板、论文搜索面板和记忆面板；后端基于 FastAPI 框架，提供 17 个 API 端点，包括 SSE 流式对话、论文搜索、会话管理和记忆接口等。前后端分别容器化部署，通过 Docker Compose 统一编排。

[图片：系统整体架构图]

后端核心为 LangGraph 构建的 Agent 引擎。其工作流程为：用户请求经 HTTP/SSE 到达 FastAPI 后端后，由 Supervisor Agent 进行意图识别，路由到对应的专业 Agent（Research / Innovator / Experiment / Chat）处理，最终由 Synthesizer Agent 汇总输出。Agent 在执行过程中可调用搜索论文、联网搜索等外部工具，并将结果存入记忆系统。

### 2.2 智能体架构

系统包含七个 Agent 节点，构成一个有向图工作流：

- **Supervisor（智能路由）**：接收用户消息，通过 LLM 进行意图分类，决定路由目标
- **Chat Agent（对话助手）**：处理普通对话和问答
- **Research Agent（文献调研）**：以 ReAct 循环模式运行，调用论文搜索和联网搜索工具完成文献调研
- **Innovator Agent（创新构思）**：基于已归档的论文信息生成创新方案并进行评估
- **Experiment Agent（实验分析）**：设计结构化的实验方案
- **Planner Agent（任务规划）**：将复杂任务分解为子任务队列
- **Synthesizer（综合输出）**：汇总各 Agent 输出，生成结构化的综合报告

[图片：Agent流水线示意图]

以一个完整的文献调研流程为例：用户提出调研需求后，Supervisor 识别出 research 意图并判断为复杂任务，路由至 Planner Agent 进行任务分解；随后多个 Research Agent 并行执行搜索任务，每个 Agent 独立调用工具并分析结果；最后 Synthesizer 读取所有子任务结果，汇总生成调研报告。

### 2.3 工具契约设计

系统实现了两种风格的工具封装。其一是基于 LangChain @tool 装饰器的传统封装，包括 arXiv 学术论文搜索工具和 Word 文档生成工具。arXiv 搜索工具支持关键词查询、结果数量控制和排序方式选择，并对 arXiv API 实施了 3 秒间隔的限流保护。Word 文档工具集包含创建文档、添加章节和添加表格三个子工具，限定在指定目录内操作以确保安全。

其二是基于 MCP（Model Context Protocol）协议的工具实现。联网搜索工具通过 DashScope MCP Streamable HTTP 协议调用阿里云百炼的搜索服务，遵循 JSON-RPC over SSE 的通信规范，工具的输入参数和输出格式均按协议定义的结构化 Schema 组织。

## 3 关键技术

### 3.1 大模型接入与配置

系统接入 DeepSeek API 作为核心大模型引擎。选择 DeepSeek 主要基于以下考虑：模型综合能力强且支持中英文，API 定价具有较高的性价比（输入约 ¥1/百万 token），兼容 OpenAI SDK 可复用 LangChain 生态，同时支持流式输出和工具调用功能。系统提供了三种模型选项：V4 Flash（快速响应，默认）、V4 Pro（更高能力）以及思考模式（显示推理链，适合复杂推理）。

API 密钥通过 .env 文件配置，不提交到版本控制系统。程序使用 python-dotenv 加载环境变量，通过 OpenAI SDK 兼容的方式调用 DeepSeek API。

### 3.2 记忆系统

系统实现了三层记忆架构。短期记忆以消息列表形式保存在会话中，支持多轮对话的上下文回溯。工作记忆以 JSON 格式持久化存储，记录论文归档、创新方案和实验设计等项目级信息。长期记忆为向量数据库预留接口，计划在未来版本中实现跨会话语义检索。

三层记忆分别由记忆管理器统一管理，各 Agent 通过记忆管理器读写对应层级的记忆数据，实现了记忆的集中管理和分散使用。

### 3.3 流式输出与前端可视化

后端采用 Server-Sent Events（SSE）协议实现流式数据推送，前端的对话消息、Agent 状态变更和工具调用信息均通过 SSE 事件流实时传输。前端实现了逐 token 渲染的流式打字机效果，同时支持 Markdown 和 LaTeX 数学公式的即时渲染。

前端 Agent 可视化面板实时展示各 Agent 的执行状态（运行中/已完成）、工具调用记录（工具名、参数、结果）以及 Agent 间的流转关系。记忆面板以可视化方式展示工作记忆中的论文归档、创新方案和实验设计等内容。

[图片：前端界面截图]

### 3.4 Docker 容器化部署

系统采用双容器部署方案。后端容器基于 python:3.12-slim 镜像，安装 Python 依赖后通过 Uvicorn 启动 FastAPI 服务。前端容器基于 node:20-alpine 镜像，构建 React 静态文件后通过 Serve 提供 HTTP 服务。Docker Compose 统一编排两个容器，配置端口映射、环境变量注入和数据卷挂载（用于会话数据持久化），通过一条命令即可完成构建和启动。

## 4 测试与验证

### 4.1 测试环境

所有测试均在 Docker 容器环境中运行，后端服务地址为 http://localhost:8001。测试前需执行 `docker compose up -d --build` 启动容器，并激活本地虚拟环境安装测试依赖。

### 4.2 核心功能测试

系统提供了 20 余个自动化测试脚本，涵盖 API 端点、Agent 流水线、前端 UI 和工具功能等多个层面。

全面功能测试覆盖了八个关键验证点：后端健康检查验证服务运行状态；arXiv 论文搜索验证工具能否正确返回结果；模型选择切换验证三种模型均可正常流式输出；Agent 流水线测试验证 Supervisor→Research→Synthesizer 的完整执行链路，包括 Agent 状态流转和工具调用记录；会话生命周期测试覆盖创建、发送消息、查询历史和删除的完整流程；Word 文档工具测试验证创建、添加章节、添加表格的功能；记忆接口测试验证读写正常。

以 Agent 流水线测试为例，系统成功触发了 9 个 Agent 事件（涉及 supervisor、research_agent、synthesizer 三个 Agent）和 12 次工具调用（包括 search_arxiv 和 web_search），Agent 状态流转正确，最终回复长度达 7629 字符，历史消息保存完整。

流式 API 测试验证了 SSE 协议的正确性，包括状态码、Content-Type、流式响应内容分段输出和消息历史保存。前端 Playwright 测试覆盖了页面加载、UI 元素存在性、聊天交互和 Agent 面板渲染等场景。[图片：测试运行截图]

### 4.3 失败路径测试

系统对多种异常场景进行了测试。对于不存在的会话 ID，系统返回空历史而非报错，实现了优雅降级。路径遍历攻击尝试被安全检查拦截并返回 404。当 LLM 无法识别用户意图时，系统自动回退到 Chat Agent 进行处理，不会崩溃。对于空消息输入，当前返回内容为空的结果，后续需增加 min_length 校验。十万字符的超大输入虽可处理但消耗大量 token，需增加长度限制。

### 4.4 安全机制验证

系统实现了多层安全防护。路径安全检查通过 realpath 比较防止路径遍历攻击；API 密钥通过 .env 文件隔离，不提交到 Git 仓库；arXiv 工具实施 3 秒请求间隔限流；LLM 调用设置 180 秒超时和 2 次自动重试；Agent 执行设 600 秒超时保护；API 输入通过 Pydantic 模型进行校验。

## 5 词元成本与安全分析

### 5.1 成本分析

DeepSeek API 的定价方案中，deepseek-chat 模型输入约 ¥1/百万 token，输出约 ¥2/百万 token，缓存命中时仅 ¥0.1/百万 token；deepseek-reasoner 模型定价相对较高。根据典型会话场景的估算，简单问答单轮成本约 ¥0.0003，文献调研（含 3 次搜索）约 ¥0.0038，综合调研约 ¥0.01，创新构思约 ¥0.004。整体而言，DeepSeek 的定价使开发测试阶段的成本极低。

### 5.2 安全与隐私

系统的安全风险主要集中在 API Key 泄露、路径遍历、提示词注入和日志敏感信息等方面。当前通过 .env 文件隔离、realpath 路径检查、系统提示词防护和 Agent 隔离等措施进行防护。在隐私保护方面，系统无用户认证机制，对话数据存储在本地 Docker volume 中，除 API 调用外数据不离开本地环境。

## 6 总结与展望

### 6.1 项目成果

本项目成功构建了一个基于多 Agent 协作的科研分析助手系统。系统规模涵盖 FastAPI 后端（17 个 API 端点）、React 前端和 LangGraph 编排的 7 个 Agent 节点。实现了真实 DeepSeek API 接入，集成了 arXiv 搜索、MCP 联网搜索、Word 文档生成等 5 个工具。记忆系统采用三层架构设计，提供 20 余个自动化测试脚本，支持 Docker 和本地开发双模式部署。

### 6.2 技术亮点

系统在以下方面体现了技术特色：LangGraph 多 Agent 编排实现了 Supervisor 智能路由、Planner 任务分解和 Synthesizer 综合输出的完整工作流；MCP 风格工具契约使工具的定义和调用规范化；三层记忆系统实现了不同粒度的信息管理；前端 SSE 流式渲染结合 Agent 可视化面板提供了良好的用户体验；完整的 Docker 容器化方案实现了便捷的部署和分发。

### 6.3 当前限制与改进方向

当前系统存在以下限制：长期记忆的向量数据库语义检索尚未集成，缺少多用户隔离和权限管理机制，Token 消耗缺乏可视化仪表板，安全防护方面输入长度限制和日志脱敏有待完善，模型选项目前仅支持 DeepSeek 系列。

未来的改进方向包括：接入 ChromaDB 实现长期记忆的语义检索；在前端增加 Token 消耗可视化面板；完善输入安全检查机制；扩展对 OpenAI、Claude 等更多模型的支持；支持 PDF 等文档上传和知识库检索；引入 JWT 认证实现用户隔离；构建评估数据集量化 Agent 回答质量。

### B. 环境变量说明

### C. 快速命令参考

Docker 启动命令为 `docker compose up -d --build`，停止命令为 `docker compose down`。本地开发时后端通过 `uvicorn backend.main:app --reload` 启动，前端通过 `npm start` 启动。运行测试使用 `python tools/test_all_features.py`，查看日志使用 `docker logs` 命令。
