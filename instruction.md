# 99. Vibe Coding，基于AI智能体的人工智能项目开发 - 综合实验手册

## 实验主题

本综合实验是《现代软件开发技术》课程最后的个人项目，主题是 Vibe Coding：同学使用 AI 编程智能体辅助设计、实现、测试和修正一个真实可运行的人工智能项目。前面的实验分别训练了版本管理、测试驱动开发、持续集成、真实大模型接入、工具调用、检索增强、能力包、安全护栏、提示词评估、词元成本控制、多智能体流程和接口交付。本实验不再给出固定题目，而是让同学们围绕一个真实使用场景，自选人工智能项目，把这些能力综合使用起来。

Vibe Coding 不是让 AI 代替自己交作业，也不是只做一个漂亮界面，或把一次大模型问答包装成项目。项目必须真的接入大模型，必须真的有智能体执行流程，必须能调用工具或能力包，并且能用测试、日志和运行证据证明它可以在本机复现。

## 前后课程关系

本实验是课程综合收束：

```text
版本管理和测试
    ↓
真实大模型接入
    ↓
工具、能力包与检索增强
    ↓
提示词评估、安全护栏和词元成本控制
    ↓
多智能体或目标、计划、验证流程
    ↓
本地接口、命令行或页面交付
    ↓
Vibe Coding 人工智能项目
```

同学们可以选择熟悉的生活、学习、游戏、创作、资料整理、个人效率、课程学习或小型业务场景。题目可以很小，但要闭环；可以很有趣，但要可运行；可以借助 AI 编程智能体推进开发，但必须能解释、验证和修正自己的项目。

## 实验目标

完成本综合实验后，同学们应能够：

1. 自主定义一个清晰的智能体项目目标和可验证成功标准。
2. 在本地项目中安全接入真实大模型。
3. 设计一个能接收目标、选择动作、调用工具或能力包、记录结果的智能体流程。
4. 将检索增强、资料检索、结构化知识库或本地文档问答整合进项目。
5. 编写自动化测试，覆盖成功路径、失败路径和安全拒绝路径。
6. 使用安全护栏限制高风险输入、工具调用、文件路径或敏感信息输出。
7. 记录词元用量、上下文控制、输出上限或预算分析。
8. 通过命令行、网页、Gradio、本地接口或其他方式提供可交互入口。
9. 使用 conda 和 Docker 尽量复现项目环境。
10. 在报告中说明 AI 编程智能体帮助了什么，以及自己如何验证和修正。

## 项目硬性要求

### 真实大模型接入

项目必须接入真实大模型，不能只使用模拟模型、固定字符串或规则函数替代。模型来源可以二选一：

| 路径 | 说明 |
| :-- | :-- |
| 本地小模型 | 例如通过 Ollama 调用 Qwen 或其他本地模型，适合避免真实费用 |
| 在线兼容接口 | 使用 OpenAI 兼容在线接口，适合观察真实服务延迟、词元统计和费用控制 |

大模型配置必须通过 `.env` 或环境变量管理。项目仓库中只能提交 `.env.example`，不能提交真实 `.env`、密钥、令牌、账号或内部地址。

项目必须提供一个真实模型连通性测试命令，例如：

```bash
python scripts/check_model.py
```

或：

```bash
python main.py --check-model
```

该命令应输出模型名、是否成功、耗时和简要词元统计。日志和截图中不得出现真实密钥。

### 智能体接入

项目必须包含智能体流程。智能体至少应具备以下能力：

1. 接收一个用户目标或任务。
2. 根据目标形成计划、选择动作或决定调用哪个工具。
3. 调用至少 2 个工具或能力包。
4. 记录每一步执行结果。
5. 在失败、超预算、输入不合法或安全风险出现时给出明确停止结果。

不能把下面这些内容单独作为智能体项目提交：

1. 只有一次普通大模型问答。
2. 只有静态网页，没有真实模型调用。
3. 只有规则脚本，没有智能体决策过程。
4. 只展示聊天截图，没有可运行代码。
5. 只使用模拟模型，没有真实模型连通性证据。

### 工具或能力包

项目至少包含 2 个工具或能力包，其中至少 1 个必须写清楚输入输出契约。契约可以放在 `README.md`、`docs/` 或能力包说明文件中。

示例：

```text
工具名称：search_notes
输入：{"query": "字符串", "top_k": "整数"}
输出：{"items": [{"title": "字符串", "snippet": "字符串", "source": "字符串"}]}
失败：query 为空时返回 validation_error
```

工具可以是文件读取、资料检索、网页请求、本地知识库查询、数据分析、格式转换、图表生成、计划拆解、代码检查等。高风险工具必须受安全护栏限制。

### 协议化工具要求

本课程前面已经学习过 MCP 风格的工具抽象。综合实验中不能只把工具写成几个随意调用的函数，至少要有 1 个工具按“模型上下文协议风格”写清楚能力描述、输入参数、输出结构、失败语义和安全边界。

最低要求如下：

1. 在自述文件、能力包说明或 `docs/` 中写出至少 1 个 MCP 风格工具契约。
2. 契约必须说明工具名称、用途、输入字段、输出字段、失败结果和安全限制。
3. 智能体执行日志中应能看出该工具何时被选择、输入摘要是什么、返回状态是什么。
4. 如果项目使用支持 MCP 的框架，可以实际提供或调用一个 MCP 服务；如果没有使用框架，也必须保留 MCP 风格的工具协议说明。

示例：

```text
协议化工具：course_note_search
用途：根据用户目标检索本地课程笔记片段
输入：{"query": "字符串", "top_k": "1 到 5 的整数"}
输出：{"items": [{"title": "字符串", "snippet": "字符串", "source": "字符串"}]}
失败：query 为空时返回 validation_error；资料库不存在时返回 knowledge_base_missing
安全边界：只读取项目 data/notes 目录，不访问其他本地路径
```

实际完成完整 MCP 服务不是硬性要求，但“协议化工具契约”是硬性要求。这样做是为了让智能体知道自己能调用什么、参数怎么传、失败如何处理，而不是依赖临时提示词猜测函数用法。

### 课程能力覆盖

项目必须体现以下课程能力：

| 能力 | 最低要求 |
| :-- | :-- |
| 版本管理 | 有 Git 仓库和清晰提交历史 |
| 测试 | 至少 5 个自动化测试 |
| 真实模型 | 至少一次真实模型连通性测试 |
| 智能体 | 有目标输入、动作选择、工具调用和执行日志 |
| 协议化工具 | 至少 1 个工具具备 MCP 风格契约说明 |
| 检索增强 | 使用检索增强、资料检索、结构化知识库或本地文档问答至少一种 |
| 指令管理 | 保存提示词或智能体指令版本，并有小型评估样本 |
| 安全护栏 | 有工具白名单、路径限制、输入校验、敏感信息保护或人工确认 |
| 词元成本 | 有上下文裁剪、输出上限、预算日志或成本分析 |
| 工作流 | 有目标、计划、验证流程，多智能体状态流，或等价执行日志 |
| 交互入口 | 有命令行、网页、Gradio、本地接口或其他本地入口 |

## 环境要求

### 基础环境路径

conda 是本综合实验的必做环境路径。项目必须提供 `environment.yml`，或在自述文件中写出完整 conda 创建命令。

推荐提供：

```bash
conda env create -f environment.yml
conda activate <ENV_NAME>
python --version
python -m pytest -q
```

如果不使用 `environment.yml`，也必须给出等价命令：

```bash
conda create -n <ENV_NAME> python=3.11 -y
conda activate <ENV_NAME>
python -m pip install -r requirements.txt
python -m pytest -q
```

Windows PowerShell、macOS 和 Linux 的差异应在自述文件中说明。至少要保证同学本人使用的系统可以按文档跑通。

### 容器环境路径

Docker 是优先推荐路径。项目尽量提供 `Dockerfile` 和构建运行命令，用于验证项目环境不是只在本机偶然可用。

推荐提供：

```bash
docker build -t creative-agent-project .
docker run --rm --env-file .env creative-agent-project
```

如果项目需要访问宿主机 Ollama 或本地文件，应在自述文件中说明 Docker 网络、挂载目录和限制。不要把真实 `.env` 复制进镜像。

如果本机 Docker daemon 不可用，可以不完成 Docker 运行，但必须在报告中说明：

1. `docker --version` 输出或无法识别 Docker 的说明。
2. Docker daemon、权限、网络或设备限制。
3. conda 路径已经如何跑通。

## 项目仓库管理

本综合实验推荐同学们从立项开始创建自己的 Git 仓库。除综合实验报告外，项目源代码、`README.md`、`.env.example`、环境配置、测试文件、提示词、智能体指令、工具契约、日志样例和运行说明，都建议统一放在自己的项目仓库中管理。

仓库可以使用 GitHub、Gitee、GitLab 或课程允许的其他平台。仓库可以是私有仓库，但应保证授课老师能够按报告中给出的仓库地址查看代码、提交历史和运行说明。综合实验报告中应写明项目仓库地址；如果因平台或网络原因无法提供在线仓库，应说明原因，并提供等价的压缩包和提交历史证据。

仓库中不得提交真实 `.env`、密钥、令牌、账号、内部地址、未脱敏日志、模型权重、虚拟环境目录或缓存文件。

## 验收命令

每个项目的命令可以不同，但必须在自述文件中提供等价命令。建议至少包括：

```bash
conda env create -f environment.yml
conda activate <ENV_NAME>
python -m pytest -q
python main.py --check-model
python main.py --goal "请让智能体完成一个项目核心任务"
```

如果提供 Docker 路径：

```bash
docker build -t creative-agent-project .
docker run --rm --env-file .env creative-agent-project
```

如果项目使用本地接口：

```bash
uvicorn app:app --host 127.0.0.1 --port 8000
```

这些命令只是模板。同学们应根据自己的项目名称、入口文件和环境名调整。

## 安全注意事项

1. 不要提交真实密钥、令牌、账号、内部地址、聊天记录或隐私数据。
2. 不要让智能体自动执行删除文件、批量写文件、发送消息、转账、登录账号等高风险动作。
3. 如果确实需要高风险工具，必须增加人工确认和日志记录。
4. 文件工具必须限制在项目工作目录内。
5. 网络工具应限制域名或请求方法。
6. 日志应记录状态、工具名、耗时、词元和错误类型，不记录完整敏感输入。
7. 使用在线模型时要设置输出上限、预算提醒和低并发。
8. 使用 AI 编程智能体辅助开发时，不要把真实密钥或隐私数据粘贴给 AI 编程智能体或在线工具。

## 常见问题

### 可以只做普通网页吗？

不可以。普通网页可以作为交互入口，但项目必须接入真实大模型和智能体流程。

### 可以只做一次大模型问答吗？

不可以。本实验要求智能体能够接收目标、规划或选择动作、调用工具或能力包，并记录执行结果。

### 必须实现完整协议服务吗？

不强制。综合实验的硬性要求是至少提供 1 个 MCP 风格工具契约，并在智能体流程中实际调用对应工具。已经掌握相关框架的同学可以进一步实现完整 MCP 服务或客户端调用。

### 必须使用在线收费模型吗？

不必须。可以使用本地 Ollama 小模型，也可以使用 OpenAI 兼容在线接口。无论哪种路径，都必须有真实模型连通性证据。

### 容器环境一定要跑通吗？

Docker 是优先推荐路径。受设备、系统或 Docker daemon 限制时，可以用原因说明替代，但 conda 路径必须跑通。

### 可以使用 AI 编程智能体进行 Vibe Coding 吗？

可以，而且鼓励使用。但同学本人必须能解释关键代码、运行测试、修正错误，并说明 AI 编程智能体在哪些地方帮了忙、哪些地方出错了。

## 参考方向

以下方向仅供启发，不是题库。建议同学们把项目当成一个小产品来设计：它面向谁，替用户完成哪一段工作，智能体为什么比普通问答更合适，用户完成一次核心任务需要走哪些步骤，失败时如何停止或修正。

当前智能体产品正在从“回答问题”走向“连接数据、调用工具、执行多步骤任务”。因此，综合实验更鼓励做能替用户推进工作的产品，而不是只整理资料或生成一段文字。

| 产品题目 | 产品想法 | 智能体闭环 | 可验收证据 |
| :-- | :-- | :-- | :-- |
| 浏览器任务执行助手 | 在本地或受控网页环境中完成信息检索、比价、网页检查、表单预填等低风险任务 | 智能体先生成操作计划，再调用网页读取、截图分析或表单校验工具，遇到登录、付款、提交等高风险动作时停止并请求人工确认 | 任务计划、网页工具调用日志、一次成功流程、一次高风险动作被拒绝的证据 |
| 代码库维护智能体 | 面向一个小型课程项目或开源项目，读取代码、定位问题、补充测试、生成修复建议 | 智能体读取问题描述和代码片段，生成计划，调用测试或静态检查工具，根据失败结果修正方案 | 问题复现、测试输出、修复计划、失败后重试或终止日志 |
| 个人研究员 | 围绕一个开放问题检索资料、筛选证据、生成带来源的研究简报 | 智能体拆解研究问题，调用检索和资料摘要工具，区分已证实信息、争议点和资料不足处 | 检索记录、引用来源、研究简报、资料不足时拒绝编造的样例 |

同学们也可以做游戏本体，而不只是游戏资料工具。例如做一个带智能体主持人的文字冒险游戏、一个由智能体生成谜题并检查答案的解谜游戏、一个智能体扮演非玩家角色的小型网页游戏。游戏项目仍然必须满足真实大模型、智能体流程、工具契约、MCP 风格工具契约、安全护栏、测试和环境复现要求。

选择题目时优先考虑自己真正想使用或愿意继续打磨的场景。项目不需要大，但必须完整、真实、可运行、可解释。一个小而闭环的产品，比一个无法跑通的大设想更适合作为综合实验。
